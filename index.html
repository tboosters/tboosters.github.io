iClass Android Mobile Device Management (MDM) System was the main project in which I researched, designed and developed a server backend, a simple frontend webpage and an Android device policy controller app during my internship period in ITWakes Ltd. The system is now under further polishing and development by my ex-colleagues (many thanks to them!) and will be released soon.

System features

  • Device Policies Management
    • Connectivity control - Block / Unblock Wifi, Bluetooth, etc.)
    • Device Security Management - Disable access to work data when device is not comply with security policies.
    • Remote wipe and lock
    • etc.
  • Remote App Management
    • Allow school admins to select app sets for individual / group of devices
    • Push app install via Google Play for Work
    • Initiate uninstall / update of apps remotely
    • etc.

Achievement

  • Completed and passed the Google EMM "Work-managed device" solution set requirement test
  • Became one of the 40 international business partners of Google in the field of Enterprise Mobility Management

Technology Stack

  • Android Device Policy Controller (DPC)
    The app performs 2 main tasks, namely device provisioning and device management: It provisions the device by exchanging information with the backend server, which delegates the provision request to Google, through HTTP requests. The device policies are then pushed to the device from the management backend by push messages, the controller decodes the message and perform respectively.
    • Programming Language: Java
    • Layouts: XML
    • External Libraries used: Google DPC Support Library, Retrofit, Google Firebase Cloud Messaging
  • Backend
    Rather than rendering webpages with dynamic content on the server, the backend mainly serves as an API server providing endpoints for the clients like the frontend and the device manager to call. This isolates the design of the frontend from the backend and allows more efficient division of labour by letting the frontend engineer to take charge. It also acts as a proxy to delegate valid user requests to the Google EMM API for app management actions.
    I also employed some kind of JSON response format standard in all API responses to make cooperation between backend and frontend easier, sample: 
                                    {
                                    "errno": 0,
                                    "msg": "Success!",
                                    "data": "{"id":1, "text":"test"}"
                                }
    • Server-side scripting: PHP5 in FuelPHP framework
    • Database: MySQL
    • External APIs used: Google Android Enterprise Mobility Management API, Google Firebase Cloud Messaging
  • Frontend
    In order to demonstrate and test the functions we implemented, I also developed a skeleton SPA frontend.
    • HTML
    • Basic styling for the SPA layout: CSS
    • AJAX requests and live data handling: Javascript
